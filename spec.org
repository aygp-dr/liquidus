* Provenance

- Spec source :: ~solidusio/solidus~, ~api/openapi/solidus-api.oas.yml~, ~master~ branch, fetched 2026-07-01.
- Format :: OpenAPI 3.0.3, hosted for docs on Stoplight (SmartBear-owned since the 2023 acquisition — not Postman; Postman is a competing platform that absorbed Fern in Jan 2026 instead). Ownership is irrelevant to this spec: Prism is MIT-licensed, standalone, no vendor account required either way.
- Surface :: 103 path templates, 33 tags (verified by ~gmake tags~ against the reified spec), two security schemes — ~api-key~ (HTTP bearer, mandatory per-operation by default) and ~order-token~ (~X-Spree-Order-Token~ header, guest-checkout scoping).
- Working invariant :: this repo never talks to a running Solidus instance. Contract in, contract out. The spec file is vendored and pinned by commit SHA at grind kickoff — a live upstream ~master~ is not a dependency.

* The actual constraint, stated precisely

"No API key" doesn't mean auth is absent from the contract — it's present and Prism enforces it. It means the CLI/TUI/MCP layer never needs a *real* admin token, because Prism's ~checkSecurity~ only validates that a bearer-shaped credential was presented, not that it's valid against a Solidus instance. A fixed stub token (~liquidus-dev-stub~) satisfies the contract permanently. This is the actual boundary being exploited, and it's worth stating as the refutation condition for the whole approach:

- *Refuted if*: Prism starts validating token authenticity against an external authority, or a build team needs to demonstrate real authz-denial behavior (403 vs 401 branching tied to role, not just presence/absence of a header). Neither is true of Solidus's ~api-key~ scheme as specified — it's presence-only at the contract level.

Second, sharper constraint, and the one actually worth the build teams' attention:

- *Prism's mock server is stateless per request.* ~POST /orders~ then ~GET /orders/{number}~ do not share state. Each response is independently generated from ~examples~ (static mode) or ~json-schema-faker~ against the schema (dynamic mode, ~-d~). Any consumer that assumes create-then-read consistency (an order that persists across the session, a cart that accumulates line items) will not get it for free.
  - *Refuted if*: a build team's demo requires a stateful checkout walkthrough (cart → address → payment → confirmation) with values that must match across steps. At that point the correct move is not "make Prism stateful" (it isn't, by design) but a thin session-shim in front of Prism that pins ~__example~ selection per the client's own in-memory state — see [[#session-shim]].

** Session shim (only if a build team hits the statefulness refutation)
:PROPERTIES:
:CUSTOM_ID: session-shim
:END:

Not built by default — named here so a team doesn't reinvent it mid-grind. A small in-process layer that intercepts SDK calls, keeps a local map of ~order_number → last-known state~, and forces Prism's response selection via the ~Prefer: example=<key>~ header (or ~__example~ query param) so a client-side "order" stays self-consistent across a demo session. This is client-side state pretending to be server state — acceptable for a fixture-only repo, wrong to ship past it.

* Component: CLI (Go)

Command surface maps directly to the spec's 33 tags (verified by ~gmake tags~ against the reified spec) rather than inventing a taxonomy — reduces spec-to-command translation to a mechanical exercise, which is the point of grinding against a contract instead of a product.

#+begin_src text
liquidus products list [--q=] [--page=] [--per-page=]
liquidus products get <id>
liquidus orders mine
liquidus orders get <number>
liquidus checkouts advance <checkout-id>
liquidus stock-locations list
liquidus taxonomies list
...  # one subtree per tag; see tags.txt for the full 29
#+end_src

- Framework: ~cobra~ for the command tree, ~liquidussdk~ for transport — no bespoke HTTP code in ~cmd/~.
- Output: ~--json~ raw passthrough is the default; a ~--table~ flag pretty-prints via ~lipgloss/table~ for the subset of tags a human is likely to inspect interactively (products, orders, stock-locations).
- Auth: reads ~LIQUIDUS_BEARER~ from env or ~--token~, defaults to the stub — never prompts for a real key, because there's nothing real behind the mock to authenticate against.

#+begin_src go :tangle cmd/liquidus/main.go
package main

import (
    "os"

    "github.com/spf13/cobra"
)

func main() {
    root := &cobra.Command{
        Use:   "liquidus",
        Short: "Solidus API surface against a Prism-mocked contract",
    }
    root.PersistentFlags().String("base-url", os.Getenv("LIQUIDUS_BASE_URL"), "Prism mock base URL")
    root.PersistentFlags().String("token", os.Getenv("LIQUIDUS_BEARER"), "bearer stub (presence-only under Prism)")

    // one AddCommand per tag subtree, generated from spec tags at scaffold time,
    // hand-edited per-command after that — do not regenerate command files on
    // every SDK bump, only the client package regenerates mechanically.
    if err := root.Execute(); err != nil {
        os.Exit(1)
    }
}
#+end_src

* Component: TUI

Same SDK, same stub auth, different surface: a slash-command palette over a list/detail split view, not a form-heavy admin clone.

- Framework: ~bubbletea~ + ~bubbles/list~ for resource browsing, a single-line command input parsed for ~/product 12~, ~/orders mine~, ~/checkout <id> advance~ style commands — mirrors the CLI's verb structure so the two don't drift into separate mental models.
- Panes: left = tag/resource tree, right = detail pane rendering the JSON shape Prism actually returned (not a hand-built form — the point is to exercise the contract, not to pre-build UI for fields that might not survive spec changes).
- No local persistence. Every ~/refresh~ re-hits Prism. This is deliberate given the statelessness constraint above — the TUI should make Prism's independence-per-request visible, not paper over it with a client cache that lies about consistency.

* Component: MCP server

Tool surface is a curated subset, not all 103 operations exposed 1:1 — an MCP tool list that size is unusable for an agent's context and defeats the purpose of grinding a clean interface.

- Transport: stdio, same ~liquidussdk~ client underneath, same stub bearer baked into the server's own env (never exposed as an MCP tool parameter — an agent should not be able to ask for a different token, since there's nothing behind the mock a different token would mean anything to).
- Tool grouping: one tool per tag for *read* operations (~list_products~, ~get_order~, ~list_taxonomies~, ...), collapsed rather than one-per-operation. *Write* operations (create/update/destroy) get individual tools with explicit schemas, since collapsing those loses the parameter contract an agent needs to call them correctly.
- Provenance: every tool call response gets wrapped with a governance tuple noting it originated from a Prism mock against a pinned spec SHA, not a live Solidus instance — an agent (or a human reading agent output later) should never mistake mock data for real store state.

#+begin_src go :tangle mcp/tools.go
package mcp

// Governance tuple attached to every tool result, not decorative —
// distinguishes contract-mock provenance from any future live-backend mode.
type Provenance struct {
    Source   string `json:"source"`    // "prism-mock"
    SpecSHA  string `json:"spec_sha"`  // pinned commit, not "master"
    Backend  string `json:"backend"`   // "none" — explicit, not implied
}
#+end_src

* Grind units

Independently gradeable, roughly ordered by dependency, not by size:

| Unit                  | Depends on   | Refutation condition                                              |

* Naming

Three candidates, in order of preference:

1. *liquidus* — the phase-diagram counterpart to solidus: the temperature boundary above which an alloy is fully molten, below which it's mushy, and *solidus* itself is the boundary below which it's fully solid. Apt because this whole toolkit operates in the pre-commitment phase — contract-only, no backend has solidified yet. Not decorative: the name states the architecture (mock-phase vs. real-phase) rather than just riffing on the parent project's name.
2. *virgule* — the actual typographic name for the ~/~ character (solidus is its Unicode name; virgule is the older term). Fits if you want the name to point at "this is about the API's shape," full stop, with no phase-diagram conceit to explain to sjg or build teams on first contact.
3. *oblique* — British/typesetting synonym for the same slash character. Same rationale as virgule, slightly less overloaded (virgule also means other punctuation marks in some traditions; oblique doesn't).

Lead recommendation: *liquidus* for the toolkit name (repo, CLI binary, TUI), with the generated client staying literally ~liquidus-sdk~ and the MCP server ~liquidus-mcp~ — keeps the phase metaphor doing real work across all three surfaces instead of being spent once and abandoned.

* Validation regime — the liquidus-00x private repos

The public repo (this one) holds the v0 spec only: ~README.org~, ~spec.org~, ~CLAUDE.md~. It never carries an implementation. Instead, private isolated repos exercise the spec against real code — each one takes only these three files as input, produces a working implementation of a single grind unit, and feeds refinements back into this spec before the next repo starts.

Ordering matters — the mocking framework must land first because everything downstream depends on a running Prism (or a validated alternative). Then the consumer surfaces in the order the user asked for (Python, Go, TS), then the MCP server.

** Feedback protocol

Each ~liquidus-00x~ repo lands with a short ~FEEDBACK.md~ enumerating what the spec left underspecified, contradicted itself on, or made harder than it should have been. The public spec revs (v0.1, v0.2, ...) accumulate those refinements — never a hidden branch, never a private consensus. The build teams are grading the spec, not the other way around.

* Customer-interaction core flow

The 33 Solidus tags are the taxonomy — but a tag list is not a flow. A customer does not think in tags. This section pins the five-stage flow a customer walks through, names which tags participate at each stage, and derives the command surface (CLI, TUI, MCP) mechanically from that mapping. Everything downstream — ~liquidus-002~ through ~liquidus-008~ — implements this surface. If the surface here does not cover a stage, the spec is wrong, not the build repo.

** Core customer flow (narrative)

Five stages, in order, from the customer's perspective (not the API's). Each stage names the Solidus tags that participate, which fixtures (F1-F5) already cover it, and the refutation condition that would invalidate the mapping.

*** Stage 1 — browse

- *Customer-observable goal*: find a product to buy.
- *Solidus tags participating*: ~Products~, ~Variants~, ~Taxonomies~, ~Taxons~, ~Option types~, ~Option values~, ~Product properties~, ~Properties~, ~Images~, ~Classifications~, ~Stock items~, ~Stock locations~.
- *Fixtures covering this stage*: F1 (products listing) and F2 (product detail by id) cover the read-side of ~Products~. No fixture yet covers taxonomy navigation or variant selection — that is a v0.1 fixture gap.
- *Refutation condition*: /If a customer cannot arrive at a specific ~variant_id~ ready to be added to a cart using only the surface named in this section, then the browse stage is wrong and MUST be updated./

*** Stage 2 — cart

- *Customer-observable goal*: assemble the intended set of line items with quantities before committing to buy.
- *Solidus tags participating*: ~Orders~ (via ~get-current-order~, the storefront "current cart" endpoint), ~Line items~ (create/update/delete against a cart), ~Coupon codes~ (apply/remove promotional codes at cart level), ~Promotions~ (read-only — the customer sees which promotions apply, not the admin config).
- *Fixtures covering this stage*: F3 partially — it covers order create, but not line-item accumulation. Fixture gap: a cart-mutation fixture (add line item, observe subtotal change) is missing from v0.
- *Refutation condition*: /If Prism's statelessness makes ~cart add~ followed by ~cart show~ return divergent line-item sets in a way F3 does not already warn about, then the cart stage is wrong and MUST be updated to cite the session-shim requirement inline./

*** Stage 3 — checkout

- *Customer-observable goal*: supply the shipping address, choose a shipping method, and arrive at a state where payment is the only remaining step.
- *Solidus tags participating*: ~Checkouts~ (advance / complete / transition, plus checkout-scoped address and line-item mutations), ~Addresses~ (canonical address shape), ~Address books~ (saved addresses for authenticated users only — guest checkout bypasses), ~Countries~ (address form field lookup), ~States~ (address form field lookup), ~Zones~ (read-only, informs shipping availability), ~Shipments~ (read-only from the customer perspective: which shipments the checkout has produced and their selected shipping method).
- *Fixtures covering this stage*: none. This is the largest fixture gap in v0. F3 stops at order-create; the ~advance-checkout~ / ~transition-checkout~ / ~complete-checkout~ triad has no acceptance fixture.
- *Refutation condition*: /If the storefront checkout flow requires a tag not enumerated here (e.g., a distinct "Shipping methods" tag surfaces in Solidus that the current 33-tag list omits), then the checkout stage is wrong and MUST be updated — and the discovery is a spec bug, not a build-repo bug./

*** Stage 4 — payment

- *Customer-observable goal*: submit payment for the completed checkout and receive a confirmed order number.
- *Solidus tags participating*: ~Payments~ (the checkout-scoped payment lifecycle: create, authorize, capture, purchase, void, credit), ~Credit cards~ (stored payment sources for authenticated users; guest checkout supplies inline), ~Store credit events~ (read-only surfacing of store credit applied to a payment).
- *Fixtures covering this stage*: none. Payment submission has no v0 fixture. The Prism statelessness constraint bites hardest here: a customer expects "payment succeeded → order number persists," and Prism will not honor that without a session shim.
- *Refutation condition*: /If a build repo cannot demonstrate payment submission and subsequent order-number retrieval within a single TUI or CLI session without invoking the session shim, then the payment stage is wrong and MUST be updated to require the shim explicitly at this stage rather than leaving it optional./

*** Stage 5 — post-order (tracking / support)

- *Customer-observable goal*: look up a placed order, see its shipment status, and (for authenticated users) review order history.
- *Solidus tags participating*: ~Orders~ (~get-order~ by number with order-token, ~list-user-orders~ for authenticated history), ~Shipments~ (order-level shipment state — /not/ carrier state; see §Explicit non-goals), ~Users~ (authenticated user profile lookup).
- *Fixtures covering this stage*: F3 covers ~get-order~ as a statelessness probe. No fixture covers ~list-user-orders~ or shipment-state retrieval. Fixture gap: an authenticated-user order-history fixture is missing.
- *Refutation condition*: /If a customer cannot retrieve an order they placed in the same session using only the order number and order token, then the post-order stage is wrong and MUST be updated (and this is almost certainly a re-firing of F3's statelessness warning at higher volume)./

** Stage-to-tag coverage summary

Of the 33 Solidus tags, 25 participate directly in the customer flow across the five stages. The remaining 8 tags (~Configuration~, ~Stores~, ~Inventory units~, ~Stock movements~, ~Customer returns~, ~Return authorizations~, and admin-only ~Users~ / ~Credit cards~ management surfaces) are admin-side and MUST NOT appear in the customer flow surface for v0 — see §Explicit non-goals.

** Cart-to-order state machine (customer view)

The customer walks through this sequence of states between adding the first line item and receiving an order number. Every state maps to a specific set of CLI/TUI/MCP commands from the surface tables above.

#+begin_src mermaid :file diagrams/liquidus-cart-to-order.svg
stateDiagram-v2
  [*] --> empty : /cart (fresh order-token)
  empty --> cart_populated : /cart add <variant> <qty>
  cart_populated --> cart_populated : /cart update | /cart remove | /coupon apply
  cart_populated --> address : /checkout advance
  address --> delivery : /checkout address (valid) → advance
  delivery --> payment_stage : /checkout advance (shipping method chosen)
  payment_stage --> confirm : /payment submit
  confirm --> complete : /checkout complete
  complete --> [*] : order number issued (post-order stage)
  cart_populated --> empty : /cart empty
  address --> cart_populated : /checkout transition cart (back)
#+end_src

The state machine MUST be readable from the customer's perspective, not the API's. Prism's statelessness makes it impossible to walk this machine end-to-end within a single mock-only session — see §Customer flow refutation on the payment-stage session-shim requirement.

* Explicit non-goals (v0/v1.0)

The following surface is deliberately excluded. Each item names why, and (where applicable) where it would live post-v1.

** Web-context "Jump" command / Command-J admin overlay

A browser-extension JS overlay in the Solidus admin panel that intercepts ~Cmd/Ctrl-J~ and exposes the CLI/TUI slash-command palette against the running admin session. Deferred as a v1+ candidate once the command surface stabilizes across grind units 002–008. Mechanical reason: the overlay requires a live Solidus instance to intercept and rewrite requests against, and the whole toolkit's foundational axiom (contract-only backend) refutes that requirement. There is no running Solidus to test against, therefore the overlay cannot be exercised in this regime. Deferred design notes MAY live in ~docs/~.

** Admin-only / CSR back-office surface

Returns processing (~Customer returns~, ~Return authorizations~), refund processing, and CSR back-office operations (~Configuration~, ~Stores~, ~Inventory units~, ~Stock movements~, admin ~Users~ management, admin ~Credit cards~ management) are excluded. The v1.0 flow is customer-facing, not CSR-facing. A CSR-facing surface is a distinct grind unit (candidate ~liquidus-009~) that MUST NOT be conflated with the customer surface — the tag overlap tempts a shared command tree that hides an auth-scope difference (customer sees own orders via order-token; CSR sees any order via api-key with admin role).

** Shipping-provider (carrier) integration

Carrier-side integrations — label generation, tracking-number lookup against UPS / FedEx / USPS APIs — are excluded. v1.0's ~Shipments~ tag exposes the customer's view of order-level shipment state (a Solidus-internal state machine), NOT carrier state. Solidus itself delegates carrier calls to plugins; modeling them in ~liquidus~ would exceed the 33-tag taxonomy.

** Stateful mock

Prism does not persist state across requests, and this spec does not ask it to. The session-shim grind unit (~liquidus-008~) is the correct home for demo-mode statefulness, not the mock server. As of v1.0 the shim is promoted from optional to REQUIRED for any consumer past the payment stage — see §Customer flow, Stage 4 refutation.

** Live-Solidus verification examples

A future v1+ artifact could ship 4–5 example snippets that talk to a *local* Solidus instance (docker-compose'd) with a real API key supplied via a git-ignored ~.env~. The natural home is a single ~docs/examples.org~ file with org-babel code blocks in shell, restclient/http, ruby, sql, and (over Emacs 32 container Tramp) direct process attachment. The file MAY be present in the public repo as a shape-of-v1 reference, but the code blocks MUST NOT be tangled or executed as part of the v0/v1.0 grind. Reason: same axiom refutation as the Cmd-J overlay — v0 through v1.0 have no running Solidus to hit, and adding one would move the repo across the ~liquidus~/~solidus~ phase boundary the name pins.

The reference file ~docs/examples.org~ is permitted but non-normative: nothing in the toolkit MUST read it, and no fixture references it. Consumers that want a live-mode should ship it themselves.

* Style guide, versioning, and fixtures

** Style guide

This spec follows the crowsnest spec style guide ([[https://wal.sh/tools/crowsnest/spec]]). Adherence points:

- Normative claims use RFC 2119 keywords in caps: MUST, SHOULD, MAY, MUST NOT.
- A single term per concept — no synonym drift (say "mock" or "Prism", not both interchangeably).
- Refutation conditions are bulleted and testable: /If <condition>, then this spec is wrong and MUST be updated./
- Numbered fixtures F1–Fn are the acceptance tests. Every grind unit MUST pass the fixtures at the spec version it consumed.

** Versioning policy

- *Major* (v0 → v1, v1 → v2): wire-breaking. A required field added or removed, an error code changed, a contract shape (URL structure, auth-model) altered. Consumers built against the prior major do not work without changes.
- *Minor* (v1.0 → v1.1 → v1.2, …): additive only. Optional fields, new fixtures, new grind units, prose clarifications. Consumers built against a prior minor within the same major continue to work.
- Every FEEDBACK.md-driven revision MUST classify itself as major or minor in the commit message and in the spec change-log table.

** Fixtures F1–F9 (v1.0)

Each fixture is a fully-specified request/response pair that a mock MUST serve and a consumer MUST be able to invoke. Fixture files live in ~liquidus-001/fixtures/~; the shapes below are the spec-level definitions.

Paths follow the reified upstream OAS (v1 Spree classic API). liquidus-001 FEEDBACK.md refuted spec v0.0's v2 storefront/platform assumption — the OAS at the pinned SHA has no such split.

- *F1 — products listing.* ~GET /products~ with bearer stub. Response MUST contain a top-level ~products~ array whose members carry ~id~, ~name~, ~display_price~. Bearer required (v1 default).
- *F2 — product detail by id + 404.* ~GET /products/{id}~. Prism does NOT distinguish known vs unknown ids — both may return 200. This IS the statelessness surprise: the v0 MUST-404-on-unknown claim is refuted. The MSW alternative can enforce 404 via hand-authored handler; Prism cannot without a session shim. Both behaviors are documented in the fixture case set.
- *F3 — order create + fetch (statelessness probe).* ~POST /orders~ then ~GET /orders/mine~ with bearer stub. Under Prism's default (stateless) mode the two calls MUST NOT be assumed to share order state; the fixture explicitly documents that the fetched order need not match the created one.
- *F4 — bearer presence enforcement.* ~GET /orders/mine~ without ~Authorization~ header MUST return 401. With any bearer-shaped value present it MUST return 200. Spec v0.0's storefront/platform split is dropped in v1.0 — the reified OAS has no such distinction.
- *F5 — governance tuple round-trip.* Any MCP tool call whose result body wraps a mock response MUST include a top-level ~provenance~ object with ~source: "prism-mock"~, ~spec_sha: <pinned SHA>~, and ~backend: "none"~. Absence is a fixture failure, not a warning.
- *F6 — order-token guest checkout* (added v1.0). A guest customer creates a cart, receives an ~order_token~, and uses it across cart-mutation calls in a single session. Under Prism this fixture is EXPECTED to fail on cross-request consistency — the failure is the point. Reveals the payment-stage session-shim requirement.
- *F7 — cart mutation* (added v1.0). ~POST /orders/<num>/line_items~ then ~GET /orders/<num>~ with order-token. MUST document that Prism's response body does NOT reflect the added line item; a session-shim consumer MUST make it reflect.
- *F8 — checkout advance* (added v1.0). ~PUT /checkouts/<num>/next~ from ~cart~ through ~complete~. Stateless Prism cannot walk the machine; MSW alternative or session shim MUST. Fixture asserts the state-machine transitions, not the mock's persistence.
- *F9 — order history for authenticated user* (added v1.0). ~GET /orders/mine~ paginated. Bearer + presumed authenticated identity. Under Prism the pagination is schema-driven; a session-shim consumer MUST return the same set across pages 1..N.

** Refutation conditions (spec-level)

Two conditions FIRED during the liquidus-001 grind (see that repo's FEEDBACK.md); the third remains untriggered. Both firings are folded into v1.0's normative shape.

- (FIRED, v0.0 → v1.0) /If a build team cannot map a Solidus API tag to exactly one CLI/TUI subcommand family, then §Component: CLI MUST be updated to name the exception and its resolution./ — Orders tag maps to two customer-flow stages (cart, post-order). Resolved in v1.0 §CLI surface by splitting into ~cart *~ and ~order *~ subcommand families.
- (FIRED, v0.0 → v1.0) /If a build team demonstrates that F3's statelessness surprise made cross-request-consistent demos impossible without shipping a session shim, then §Session shim MUST be promoted from optional to required for that consumer./ — Fired even at the mocking-framework grind. Resolved in v1.0 §Session shim (implicit under Customer flow Stage 4 payment).
- (NOT FIRED) /If Prism starts validating bearer authenticity against an external authority, or a build team needs role-based authz-denial behavior (403 vs 401 tied to role), then this spec is wrong and MUST be updated to describe a stateful-auth mode./ — Prism's ~checkSecurity~ remains presence-only. Stub bearer still satisfies the contract.
